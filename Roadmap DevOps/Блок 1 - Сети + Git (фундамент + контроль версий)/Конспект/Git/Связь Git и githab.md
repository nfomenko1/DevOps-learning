## Как вообще связаны Git и GitHub

- **Git** — программа на твоём ПК, которая ведёт историю файлов (коммиты, ветки и т.д.).[](https://docs.altlinux.org/ru-RU/archive/7.0.5/html/simply-linux/ch39s02s03.html)​
    
- **GitHub** — сайт, где можно хранить копию твоего репозитория и делиться им (remote).[](https://docs.github.com/ru/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/about-readmes)​
    
Связка выглядит так:
1. Локально в Git Bash:
    
    - Создаёшь/меняешь файлы.
        
    - Делаешь `git add` → `git commit` → это фиксирует историю **у тебя на диске**

> Git и GitHub — разные вещи:
- `git add`, `git commit` работают **только локально**, на твоём компьютере.
    
- `git push` отправляет уже сохранённые локальные изменения на GitHub.



## Что значит «working directory» и «пойдут в коммит»

Представь git как фотоаппарат:

- **Working directory** — всё как сейчас в папке: файлы, которые ты меняешь. Git это «видит», но ещё не сфотографировал.
    
- **Коммит** — фотография (снимок) всей папки в данный момент времени + комментарий. Это и есть «история».
    

Между ними есть «корзина перед фотиком»:

- **`git add`** — ты кладёшь файлы в «корзину» (staging area): «эти готовь к фотке».
    
- **`git commit`** — фотик щёлкнул: всё, что было в корзине, попало на снимок (в историю).
    

Поэтому:

- «изменения есть в working directory, но git их только видит» — ты поменял файл, git показывает его в `git status` как _modified_ или _untracked_, но он ещё **не попадёт** в следующий `git commit`, пока ты не сделаешь
